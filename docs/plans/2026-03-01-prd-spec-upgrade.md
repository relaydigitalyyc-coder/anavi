# ANAVI PRD + Spec Upgrade (Whitepaper‑Aligned)

**Date:** 2026-03-01  **Source Inputs:** Strategic White Paper (Jan 2026), TL;DR White Paper, Master PRD MVP‑1 (Feb 2026), current repo state (code + TODO).  **Purpose:** Consolidate whitepaper → MVP‑1 requirements into a single actionable PRD + spec layer, clarify scope, and define acceptance criteria.

## 1) Product Thesis (Whitepaper‑Aligned)
ANAVI is a **Relationship Operating System** for private markets. It replaces broker‑chain opacity and fraud with **verified identity**, **relationship custody**, **blind matching**, and **automated economics**. Core differentiator: relationships become **timestamped, attributed assets** with lifetime value.

**Whitepaper pillars (non‑negotiable):**
1. Verified Identity & Trust Scoring
2. Relationship Custody (core innovation)
3. AI‑Powered Blind Matching
4. Embedded Deal Infrastructure (Deal Rooms)
5. Transparent Economics & Automated Payouts

---

## 2) MVP‑1 Definition (PRD‑Aligned)
MVP‑1 is a **minimum defensible product** (not minimal). It must close founding members and run pilot deals without apologizing for missing core value.

### Must‑Ship (MVP‑1)
- **Relationship Custody:** upload, **RFC‑3161 timestamp**, custody receipt, portfolio view
- **Verification:** Tier 1–2 flow + human review + OFAC/PEP baseline
- **Intent Creation:** structured buy/sell/invest/raise intents + manual matching assist
- **Match Notifications:** in‑app + email
- **Deal Rooms:** NDA generation, doc sharing, e‑signature, basic audit log
- **Onboarding:** 3 personas, full 5‑stage flow, first value moment
- **Demo Mode:** 4 persona library, guided demo tour, apply CTA
- **Tooltip System:** 5 tooltip types (concept/action/status/data/empty state)
- **Tours:** onboarding tour (7 steps) + demo tour (6 steps)
- **Payout Dashboard:** attribution history and payout status (manual payment ok)
- **Mobile Responsive:** core flows usable on 375px

### Out of Scope (Phase 2+)
- Full AI matching engine (train on Phase 1 closed‑deal data)
- Relationship Intelligence Graph + Intent Market Depth
- AI diligence copilot + ANAVI Intelligence product
- Automated payment rails + Compliance Passport API
- Portfolio tokenization + Trust Inheritance Protocol

## 3) Functional Requirements (Condensed)

### 3.1 Relationship Custody
**Goal:** A relationship upload produces a legally defensible priority claim.

**Requirements:**
- Upload a relationship contact (identity + minimal proof)
- Generate RFC‑3161 timestamp + custody receipt
- Relationship is encrypted at rest (zero‑knowledge custody concept)
- Blind by default; requires mutual consent for disclosure
- Store attribution chain for lifetime payouts

**Acceptance Criteria:**
- Custody receipt includes timestamp, hash, originator ID, reference ID
- Relationship disclosure requires explicit consent flow
- Relationship appears in portfolio view with custody status

### 3.2 Verification & Trust
**Goal:** Network has baseline trust barrier and dynamic trust score.

**Requirements:**
- Tier 1: identity + business registration + OFAC/PEP
- Tier 2: doc upload + AML questionnaire + human review
- Trust Score: visible score + breakdown + update on key events

**Acceptance Criteria:**
- User can complete Tier 1 onboarding flow
- Verification status + trust score visible on dashboard and profile

### 3.3 Intent & Matching
**Goal:** Structured intents can be matched while preserving identity.

**Requirements:**
- Intent types: buy/sell/invest/raise/co‑invest
- Structured fields + optional notes
- Manual match workflow to simulate early matching engine
- Match notifications + mutual‑interest handshake

**Acceptance Criteria:**
- Intent → match → mutual consent → deal room flow works end‑to‑end

### 3.4 Deal Rooms & Audit
**Goal:** A secure deal room is created post‑consent.

**Requirements:**
- Auto NDA generation per jurisdiction
- Document uploads with versioning
- E‑signature hook (mock ok) + audit trail entry

**Acceptance Criteria:**
- Deal room has docs tab, NDA signed state, audit entries

### 3.5 Economics & Payouts
**Goal:** Attribution is visible and deterministic.

**Requirements:**
- Originator share 40–60% (configurable per deal)
- Payout dashboard shows attribution chain + status

**Acceptance Criteria:**
- Deal participant attribution stored and visible
- Payout dashboard displays expected payouts

1. **Weeks 1–3:** custody proof + verification provider integration + RFC‑3161 timestamp
2. **Weeks 2–5:** intent flow + manual matching workflow + match notifications
3. **Weeks 4–7:** onboarding 3 personas + trust score UX + dashboard polish
4. **Weeks 6–9:** demo mode + tooltip system + guided tours
5. **Weeks 8–11:** deal room docs + NDA + e‑sign hook + payout dashboard
6. **Week 12:** stabilization, bug bash, pilot cohort readiness

## 8) Decisions & Tradeoffs
- MVP uses **manual matching assist** to avoid premature AI accuracy risk.
- Security hardening is deferred, but custody proof and audit trails **must** be credible.
- Demo mode is a sales asset; it is treated as core, not optional.
